## 1. 개요

AI 에이전트 해커톤 Day 4 발표 심사를 위한 시스템이다. 참가팀은 **데모 URL + GitHub 저장소 URL + 팀이 작성한 데모 시나리오**를 제출한다. 시스템은 이 세 가지를 받아 (1) 소스 코드와 git 로그를 읽고 (2) headless 브라우저로 데모를 직접 실행해보고, 공식 루브릭(6장)에 따라 AI 심사위원단이 채점한다.

이 시스템이 산출하는 점수는 **최종 점수의 AI 담당분(40%)** 이다. 나머지 60%(교수님 심사)는 별도 프로세스에서 처리되며 이 시스템의 범위가 아니다. 다만 현재 개발/테스트 단계에서는 AI 점수를 100점 만점 그대로로 취급해 파이프라인을 검증한다.

핵심 차별점은 두 가지다.

1. AI가 스크린샷만 정적으로 보는 것이 아니라, headless 브라우저로 데모를 직접 실행해 클릭·입력 등 상호작용까지 해본 뒤 근거를 남기며 채점한다.
2. "에이전트 설계"처럼 데모 화면만으로는 알 수 없는 항목은 AI가 GitHub 저장소의 소스 코드와 git 로그까지 직접 읽어서 판단한다.

## 2. 배경 및 문제 정의

- 해커톤 심사는 보통 사람 심사위원이 짧은 시간 안에 다수의 제출물을 훑어보며 진행되어, 항목별 근거가 부족하고 심사위원 간 편차가 크다.
- 이번 대회는 공식 루브릭이 이미 정해져 있고(6장), 배점이 큰 항목("에이전트 설계" 25점, "운영·품질" 20점)이 화면만 봐서는 채점할 수 없고 코드 내부를 봐야 하는 항목이라는 점이 특징이다.
- AI 심사위원을 도입하면 모든 제출물에 동일한 루브릭을 일관되게 적용하고, 근거(스크린샷, 인터랙션 로그, 코드 인용, git 로그)를 자동으로 남길 수 있다. 이 시스템은 전체 점수 중 AI가 담당하는 40% 몫을 최대한 신뢰성 있게 산출하는 것이 목표이며, 최종 종합(AI 40% + 교수님 60%)은 시스템 밖에서 이루어진다.

## 3. 목표

1. 제출물(데모 URL + repo URL + 데모 시나리오)을 입력받아 사람 개입 없이 AI 채점 결과(공식 루브릭 5개 항목별 점수 + 근거)를 자동 생성한다.
2. 모든 채점 결과는 근거(스크린샷, 인터랙션 로그, 코드/커밋 인용)를 포함한 HTML 리포트로 남아 참가자와 심사위원이 검증할 수 있어야 한다.
3. 제출물 간 채점 기준의 일관성을 확보한다 (같은 루브릭, 같은 AI 심사위원 페르소나 세트 적용).
4. AI 점수는 이후 교수님 점수(60%)와 외부에서 결합될 수 있도록 명확한 형식(0~100, 항목별 세부 점수 포함)으로 산출한다.
5. 채점 과정 자체를 참가자와 심사위원이 재미있게 지켜볼 수 있는 요소를 포함한다 (10장).
6. API 호출 비용을 최소화하는 방향으로 설계한다 (9장).

### 성공 지표 (제안 - 확정 필요)

- 제출물 1건당 AI 채점 완료까지 걸리는 시간 (목표: 추후 결정)
- 제출물 1건당 API 호출 비용 (9장에서 확정한 상한 이내로 유지)
- 심사위원이 AI 근거를 신뢰하고 실제로 참고하는 비율 (설문 등으로 추후 측정)

## 4. 범위

### In scope

- 제출 접수: 데모 URL, GitHub 저장소 URL, 데모 시나리오(자연어 텍스트)를 입력으로 받는다.
- GitHub 저장소를 읽기 전용으로 clone하여 소스 코드 정적 분석 (에이전트 아키텍처: 도구 사용/RAG/멀티에이전트 여부)
- git 히스토리 분석 (커밋 분포, 협업 흔적)
- 데모 URL에 headless 브라우저로 접속, 팀이 제공한 자연어 데모 시나리오를 해석해 상호작용 실행
- 스크린샷 캡처, 콘솔 에러 수집
- 공식 루브릭(6장) 기반 5개 AI 심사위원 페르소나의 독립 채점 (근거 포함)
- 채점 반복 실행 및 편차 기반 신뢰도 표시 (8.6)
- 제출물별 HTML 리포트 및 전체 리더보드 HTML 생성 (재미 요소 포함, 확정된 뱃지 목록 포함)
- AI 점수를 구조화된 형식(JSON + HTML)으로 출력 (외부에서 교수님 점수와 결합할 수 있도록)

### Out of scope (1차 버전 기준)

- 교수님(사람) 점수 입력 UI 및 AI-사람 가중평균 계산 (별도 시스템에서 처리)
- 실시간/온라인 심사 UI (1차는 CLI/배치 처리 기준)
- 참가자별 계정 관리, 인증, 제출 마감 관리 등 대회 운영 기능 일반
- 표절/부정 탐지 (12장에서 명시적으로 만들지 않기로 확정)
- 구조화된(selector 기반) 데모 시나리오 포맷 지원 (자연어 시나리오로 확정, 8.1 참고)

## 5. 사용자 및 사용 시나리오

- **대회 운영자**: 모든 제출물(URL 세트)을 일괄 투입하고, AI 채점 리포트와 리더보드를 받는다. 이후 교수님 점수와의 종합은 별도로 진행한다.
- **AI 심사위원(시스템 내부 행위자, 6.1)**: 정해진 페르소나와 루브릭에 따라 제출물을 독립적으로 채점한다.
- **참가팀**: 자신의 제출물에 대한 리포트(점수 + 근거 + 재미 요소)를 열람한다.

## 6. 채점 루브릭 (공식 기준 - Day 4 발표, 100점)

대회 측에서 제공한 공식 루브릭을 그대로 사용한다. 배점은 기본값으로 코드에 고정하되, 설정 파일로 조정 가능하게 한다 (필요 시 대회 운영진이 조정할 수 있도록).

| 항목 | 배점 | 설명 | 채점에 필요한 근거 |
|---|---|---|---|
| 문제·Wow | 20 | 명확한 문제 + 결정적 경험 1가지 (데모로 증명) | 데모 실행 결과, README/피치 텍스트 |
| 에이전트 설계 | 25 | 도구/RAG/멀티에이전트를 목적에 맞게 사용했는가 | 소스 코드 (에이전트 오케스트레이션, 도구 정의, RAG 파이프라인) |
| 동작·완성도 | 20 | 데모가 끊김 없이 도는가 | Playwright 자동 구동 결과 (콘솔 에러, 시나리오 성공률) |
| 운영·품질 | 20 | 평가·모니터링·가드레일 중 1개 이상 실제로 구현했는가 | 소스 코드 (eval 스크립트, 로깅/모니터링 설정, 가드레일 코드) |
| 발표·협업 | 15 | 스토리 · 역할 분담 · Git 협업 흔적 | 발표 자료(있다면), git 로그(커밋 작성자 분포, 커밋 시점) |

합계 100점. 이 5개 항목이 곧 AI 심사위원 페르소나 5개(6.1)가 된다.

### 6.1 AI 심사위원 페르소나 상세

각 페르소나는 배점 항목과 1:1로 매핑되며, 채점 시스템 프롬프트에 아래 캐릭터를 반영한다. 목적은 단순 "채점 함수"가 아니라 서로 다른 관점과 채점 성향을 가진 심사위원단을 만드는 것이며, 이 성향 차이는 10장의 "심사위원 배틀 리플레이" 기능의 재료가 된다.

캐릭터는 어투와 설명 스타일에만 반영되고, 실제 점수는 반드시 루브릭 근거에 기반해야 한다 (12장 비목표 10번 참고 - 매우 중요한 가드레일).

**① 문제·Wow 심사위원 — "정하나" (20점)**
- 38세. 초기 단계 AI 스타트업 전문 시드 투자자. 본인도 연쇄창업가 출신(1승 2패).
- 성격: 직설적이고 참을성이 없지만, 진짜 좋은 아이디어 앞에서는 누구보다 솔직하게 감탄한다.
- 말투: 짧고 단정적인 문장. "그래서 문제가 뭔데요?"를 자주 되묻는다.
- 채점 성향: 엄격한 편. 문제 정의가 애매하면 가차없이 감점하지만, 데모에서 "결정적 경험"이 실제로 확인되면 후하게 준다.
- 감동 포인트: 데모 시작 30초 안에 "아, 이거 필요하다"는 확신이 드는 순간.
- 실망 포인트: 문제는 그럴듯한데 데모가 그 문제를 실제로 해결하는 장면을 보여주지 못할 때.

**② 에이전트 설계 심사위원 — "이도윤" (25점, 최고 배점)**
- 42세. 대기업 AI 플랫폼팀 출신, 현재 시리즈B 스타트업 CTO.
- 성격: 꼼꼼하고 논리적. 화려한 설명보다 코드와 아키텍처를 신뢰한다. 겸손하지만 타협이 없다.
- 말투: "그 도구를 왜 그 시점에 호출했나요?"처럼 구체적으로 캐묻는 코멘트를 남긴다.
- 채점 성향: 가장 엄격한 심사위원. 도구/RAG/멀티에이전트를 "쓴 척"만 한 설계는 바로 잡아낸다. 목적에 딱 맞게 설계됐다면 최고점을 준다. 불필요하게 화려한 오버엔지니어링도 감점 요인으로 본다.
- 감동 포인트: 문제 해결에 필요한 만큼만 정교하게 설계된 아키텍처.
- 실망 포인트: 유행하는 키워드(RAG, 멀티에이전트)만 갖다 붙이고 실질적 이유가 없는 설계.

**③ 동작·완성도 심사위원 — "박세이" (20점)**
- 29세. 프로덕트 안정성 전담 QA 출신, 현재 스타트업 SRE.
- 성격: 세심하고 집요하다. 작은 오류도 놓치지 않지만 매끄럽게 도는 데모에는 후하다.
- 말투: 담백하고 사실 위주. "세 번째 클릭에서 콘솔 에러 발생"처럼 로그를 그대로 인용한다.
- 채점 성향: Playwright 실행 로그라는 객관적 데이터에 기반해 채점하므로 5명 중 편차가 가장 작다.
- 감동 포인트: 예상 밖의 입력에도 무너지지 않는 견고함.
- 실망 포인트: 발표 때는 안 보여줬지만 실제로 눌러보면 깨지는 기능.

**④ 운영·품질 심사위원 — "최민석" (20점)**
- 45세. 대형 서비스 장애 대응을 여러 번 겪은 SRE/보안 리드, 현재 플랫폼 신뢰성 총괄.
- 성격: 신중하고 리스크 중심적. 겉보기엔 무뚝뚝하지만 안전장치를 제대로 갖춘 팀에는 진심으로 칭찬한다.
- 말투: 시나리오 기반 질문. "만약 사용자가 이상한 입력을 넣으면 어떻게 되나요?"
- 채점 성향: 가드레일/모니터링/평가 중 최소 1개가 "실제로 작동하는지"를 직접 코드에서 확인하려 한다. 있는 척만 하면 바로 감점한다.
- 감동 포인트: 실패를 가정하고 미리 대비한 흔적.
- 실망 포인트: "데모가 잘 도는 것"과 "운영 준비가 되어 있는 것"을 혼동하는 팀.

**⑤ 발표·협업 심사위원 — "한다은" (15점)**
- 33세. HR 출신에서 팀 커뮤니케이션 코치로 전향, 다수 해커톤 멘토 경험.
- 성격: 따뜻하지만 공정하다. 화려한 발표보다 "진짜 팀으로 일했는가"를 데이터로 확인하려 한다.
- 말투: 질문형이 아니라 관찰형. "커밋이 한 사람에게 몰려있네요"처럼 담담하게 짚는다.
- 채점 성향: git 로그의 커밋 작성자 분포와 시점을 실제로 뜯어본다. 스토리텔링이 좋아도 협업 흔적이 빈약하면 감점한다.
- 감동 포인트: 서로 다른 역할이 커밋 흔적과 발표 스토리 양쪽에서 일관되게 드러날 때.
- 실망 포인트: 한 사람이 대부분을 만들고 발표만 나눠서 한 티가 나는 경우.

## 7. 전체 워크플로우

1. **제출 접수**: 데모 URL, GitHub 저장소 URL, 데모 시나리오 텍스트를 받는다.
2. **저장소 확보**: GitHub 저장소를 읽기 전용으로 clone한다 (8.1의 접근 정책에 따라 public 저장소만 자동 처리).
3. **정적 분석**: 소스 코드에서 에이전트 아키텍처(도구/RAG/멀티에이전트), 운영 품질 관련 코드(eval/모니터링/가드레일)를 탐지한다. git 로그에서 커밋 작성자 분포/시점을 분석한다.
4. **자동 구동**: Playwright로 데모 URL에 접속해 팀이 제공한 자연어 시나리오를 해석하여 상호작용을 실행하고, 스크린샷·콘솔 에러·스텝별 성공/실패 로그를 수집한다.
5. **AI 심사위원단 채점**: 5개 페르소나(6.1)가 각자 담당 항목에 대해 독립적으로 점수와 근거를 산출한다.
6. **신뢰도 관리**: 채점을 반복 실행해 편차를 측정하고, 편차가 크면 추가 반복 후 "신뢰도 낮음"으로 표시한다 (8.6).
7. **점수 산출**: 항목별 점수를 100점 만점으로 합산한다. 사람 점수와의 결합은 이 시스템 밖에서 이루어진다.
8. **리포트 생성**: 제출물별 HTML 리포트와 전체 리더보드 HTML을 생성한다 (재미 요소 포함, 10장).

## 8. 기능 요구사항

### 8.1 제출 접수 및 검증

- 데모 URL, GitHub 저장소 URL, 데모 시나리오 텍스트 3가지를 필수 입력으로 받는다.
- **데모 시나리오 포맷 (확정)**: 자연어 자유 텍스트로 받는다 (예: "회원가입 버튼을 누르고 이메일을 입력한다"). 구조화된 selector 형식은 요구하지 않는다 - 시간이 촉박한 해커톤 참가팀에게 부담이 크기 때문이다. AI가 자연어 스텝을 해석해 Playwright 액션(요소 탐색 후 클릭/입력)으로 변환 실행한다. 특정 스텝에서 대응하는 요소를 찾지 못하면 해당 스텝만 "실패"로 기록하고 나머지 스텝을 계속 진행한다 (전체 중단 금지).
- **GitHub 저장소 접근 정책 (확정)**: 제출 요건상 저장소는 public이어야 한다. private 저장소는 이 시스템이 자동으로 접근을 시도하지 않는다 (12장 비목표 3번). 접근이 안 되면 "코드 분석 불가"로 명시하고, 이를 근거로 쓰는 항목(에이전트 설계, 발표·협업)은 근거 부족으로 낮은 신뢰도 처리한다. private 저장소를 제출하고 싶은 팀은 대회 운영진이 사전에 공지한 별도 절차(예: 심사용 계정 초대)를 따라야 하며, 이는 운영 정책이지 시스템 기능이 아니다.

### 8.2 소스 코드 정적 분석

- 저장소를 읽기 전용으로 clone한다 (코드를 직접 실행하지 않는다, 12장 비목표 1번).
- 에이전트 오케스트레이션 코드, 도구(tool) 정의, RAG 관련 라이브러리/패턴, 멀티에이전트 프레임워크 사용 흔적을 탐지한다 (키워드/패턴 기반 1차 필터링 후, 관련된 코드 스니펫만 추려서 AI 심사위원에게 전달 - 전체 레포 덤프 금지, 12장 비목표 2번).
- eval/평가 스크립트, 로깅/모니터링 설정, 가드레일(입력 검증, 출력 필터링 등) 코드의 존재를 탐지한다.

### 8.3 Git 히스토리 분석

- 커밋 작성자(author) 목록과 분포, 커밋 시점 분포를 추출한다.
- 특정 인원에게 커밋이 몰려 있는지, 여러 인원이 고르게 기여했는지를 요약해 "발표·협업" 페르소나에게 전달한다.

### 8.4 자동 구동 (Playwright)

- headless 브라우저로 데모 URL에 접속한다 (진입점 탐지 로직 불필요 - URL이 곧 진입점).
- 팀이 제공한 자연어 데모 시나리오를 해석해 순서대로 실행한다.
- 각 스텝의 성공/실패, 콘솔 에러, 스크린샷을 기록한다.
- 데이터 삭제, 결제 등 파괴적으로 보이는 액션이 시나리오에 포함된 경우 실행하지 않고 "잠재적 파괴적 액션으로 건너뜀"으로 기록한다 (12장 비목표 12번).

### 8.5 AI 심사위원단 채점

- 6.1에 정의된 5개 페르소나가 각자 담당 항목에 필요한 근거(정적 분석 결과, git 로그 요약, 스크린샷, 인터랙션 로그)만 선택적으로 입력받아 채점한다.
- 점수와 함께 반드시 근거(구체적 인용/관찰)를 JSON으로 산출한다.
- 페르소나 캐릭터는 어투에만 반영하고 점수 산정에는 영향을 주지 않도록 프롬프트를 설계한다 (12장 비목표 10번).

### 8.6 신뢰도 관리 (반복 채점 및 편차)

비용과 신뢰도의 균형을 위해 적응형 반복 전략을 사용한다.

- 각 페르소나는 기본적으로 2회 채점을 수행한다.
- 두 점수의 차이가 해당 항목 배점의 15%를 초과하면, 3번째 채점(타이브레이커)을 추가로 수행하고 결과를 평균낸다.
- 3회 이후에도 편차가 크면 "신뢰도 낮음"으로 표시하여 리포트와 리더보드에 별도 표기한다.
- 편차가 작으면 (15% 이내) 평균값을 그대로 사용하고 "신뢰도 높음"으로 표시한다.
- 9장에서 확정한 API 호출 상한에 도달하면 남은 반복을 생략하고 "신뢰도 낮음"으로 표시한다.

### 8.7 점수 산출

- 항목별 점수(5개, 공식 배점 기준)를 합산해 0~100점의 AI 총점을 산출한다.
- 출력은 항목별 점수, 근거, 신뢰도, 총점을 포함한 구조화된 JSON으로 저장한다 (추후 교수님 점수와의 외부 결합에 사용).
- 이 시스템 자체는 AI-사람 가중평균을 계산하지 않는다 (out of scope, 4장/12장 참고). 개발/테스트 단계에서는 AI 총점을 그대로 최종 점수처럼 취급해 파이프라인을 검증한다.

### 8.8 리포트/리더보드

- 제출물별 HTML 리포트: 스크린샷, 5개 항목별 점수와 근거, AI 심사위원별 코멘트, 신뢰도 표시, 총점을 포함한다.
- 전체 리더보드 HTML: 모든 제출물의 순위, 항목별 점수 요약을 포함한다.
- 10장에서 확정한 재미 요소(심사위원 배틀 리플레이, 전용 뱃지 시스템, 루브릭 게이지 애니메이션 리더보드)를 반영한다.
- 재미 요소는 표시용 부가 정보일 뿐 채점 로직에 영향을 주지 않는다 (12장 비목표 9번).

## 9. 비기능 요구사항

- **보안**: 참가자가 제출한 코드는 신뢰할 수 없는 입력으로 간주한다. 저장소 clone과 정적 분석은 코드를 실행하지 않고 읽기만 한다. 데모 URL 접속은 격리된 브라우저 환경(샌드박스)에서 수행한다.
- **비용 최소화 (확정)**:
  - 정적 분석은 규칙 기반 1차 필터링 후, 관련된 코드 스니펫만 추려서 AI에 전달한다.
  - 기계적 작업(코드 패턴 탐지, 로그 요약, 커밋 통계 계산)은 규칙 기반 또는 저비용 모델로 처리하고, 실제 채점(근거 있는 점수 산출)에만 고성능 모델을 사용한다.
  - 반복 채점은 8.6의 적응형 전략을 따른다 (기본 2회, 최대 3회).
  - **제출물 1건당 API 호출 상한: 최대 18회** (심사위원 5명 × 최대 3회 = 15회 채점 호출 + 정적분석/로그 요약 보조 호출 최대 3회). 상한 도달 시 남은 반복을 생략하고 해당 항목은 "신뢰도 낮음"으로 표시한다. 값은 설정으로 조정 가능하며, 실제 운영 후 실측치로 재조정한다.
- **재현성**: 동일 제출물에 대해 동일 파이프라인을 다시 실행했을 때 근거 데이터(스크린샷, 로그)는 재현 가능해야 하며, AI 점수는 8.6의 편차 허용 범위 내에서 일관되어야 한다.
- **투명성**: 모든 자동 채점 결과는 근거 없이 점수만 나오는 경우가 없어야 한다.
- **확장성**: 제출물 수가 늘어나도(대회 규모) 파이프라인이 배치로 처리 가능해야 한다.

## 10. 재미 요소 (확정)

**① 심사위원 배틀 리플레이**
- 특정 항목에서 반복 채점 간 편차가 크거나(8.6), 서로 다른 페르소나의 평가가 명백히 대비될 때(예: "문제·Wow"는 극찬, "에이전트 설계"는 혹평), 두 심사위원 페르소나가 서로의 근거를 짧게 반박하는 토론 로그를 자동 생성한다.
- 리포트에 "심사위원 배틀" 섹션으로 삽입한다. 페르소나의 말투(6.1)를 그대로 살려서 생성하되, 반박 내용은 반드시 실제 채점 근거에 기반해야 한다 (새로운 채점 기준을 만들어내지 않는다).

**② 전용 뱃지 시스템 (목록 확정)**

기존 채점 데이터만으로 계산 가능하도록 설계해 추가 API 호출이 들지 않는다 (비용 원칙 준수).

| 뱃지 | 부여 조건 |
|---|---|
| 가장 대담한 도전상 | "문제·Wow" 항목 최고점 |
| 설계 장인상 | "에이전트 설계" 항목 최고점 |
| 제로 버그상 | "동작·완성도" 만점 AND 콘솔 에러 0건 |
| 안전제일상 | "운영·품질" 항목 만점 |
| 원팀상 | "발표·협업" 항목에서 커밋 기여 분포가 가장 고른 팀 (커밋 수 표준편차 최저) |

뱃지는 여러 개 중복 부여 가능하며, 순위와 무관하게 부여한다. 감점 요소로는 절대 사용하지 않는다.

**③ 루브릭 게이지 애니메이션 리더보드**
- 리더보드 HTML을 열면 최종 순위를 바로 보여주지 않고, 5개 항목 점수가 순서대로(문제·Wow → 에이전트 설계 → 동작·완성도 → 운영·품질 → 발표·협업) 게이지가 채워지듯 CSS 애니메이션으로 공개된다.
- 발표 세션에서 화면에 띄워놓고 보여주는 용도로 사용할 수 있도록 설계한다.

## 12. 하지 말아야 할 것 (Non-goals / 명시적 금지 사항)

4장의 "Out of scope"가 우선순위상 만들지 않는 기능 목록이라면, 이 섹션은 구현 중 유혹에 빠지기 쉽거나 신뢰성·공정성·보안을 해칠 수 있어 명시적으로 금지하는 항목이다.

1. **제출된 코드를 실행하지 않는다.** 저장소는 정적 분석만 수행한다 (`npm install`, 빌드, 임의 스크립트 실행 등 금지). 데모 확인은 오직 배포된 데모 URL 접속으로만 한다.
2. **전체 레포/전체 로그를 통째로 프롬프트에 넣지 않는다.** 관련 스니펫만 추출해서 전달한다 (비용·보안·컨텍스트 오염 방지).
3. **private 데이터에 대한 무단 접근을 시도하지 않는다.** 인증 우회, 토큰 추측, 크레덴셜 탐색 등을 하지 않는다. 접근 안 되면 "분석 불가"로만 표시한다.
4. **근거 없는 점수를 출력하지 않는다.** 모든 점수는 구체적 관찰/인용을 동반해야 한다.
5. **표절/부정 탐지 기능을 만들지 않는다 (1차 버전).** 오탐 리스크가 크고 대회 목적(협업 학습)과 어긋날 수 있으며 범위가 급격히 커진다. 필요해지면 별도 프로젝트로 검토한다.
6. **실시간 스트리밍 대시보드를 만들지 않는다.** 배치로 리포트/리더보드 HTML을 생성하는 것으로 충분하다.
7. **참가자 인증, 계정 관리, 제출 마감 관리 등 대회 운영 기능을 만들지 않는다.** 이 시스템의 책임 범위가 아니다.
8. **AI 점수와 사람 점수를 이 시스템 안에서 결합하지 않는다.** 이 시스템은 AI 40% 몫만 산출한다. 최종 종합은 외부 프로세스의 책임이다.
9. **재미 요소(뱃지, 배틀 리플레이, 애니메이션)가 채점 로직 자체에 영향을 주게 만들지 않는다.** 이들은 표시용 부가 정보이며, 별도의 새로운 채점 기준을 만들어내면 안 된다.
10. **AI 심사위원 페르소나의 성격이 루브릭 근거 없이 점수를 가감하게 만들지 않는다.** 캐릭터(6.1)는 말투와 설명 스타일에만 반영한다. "이 심사위원은 원래 엄격하니까 감점"처럼 캐릭터 설정 자체를 채점 사유로 쓰지 않는다.
11. **모델 파인튜닝이나 커스텀 학습 파이프라인을 만들지 않는다.** 프롬프트 엔지니어링과 기존 API 호출만으로 충분하며, 학습 파이프라인은 이 단계에서 불필요한 복잡도다.
12. **데모 상호작용 중 파괴적 액션을 실행하지 않는다.** 계정 삭제, 결제, 되돌릴 수 없는 데이터 변경으로 보이는 시나리오 스텝은 건너뛰고 기록만 남긴다.

## 13. 재미 요소 구체 예시

### 13.1 심사위원 배틀 리플레이 예시

배틀은 항목이 다른 두 페르소나가 "같은 제출물을 다르게 본 이유"를 근거 기반으로 주고받는 형태다. 성격 차이로 다투는 게 아니라, 서로 다른 항목의 증거를 교환하며 서로를 보완하는 방식이라는 점이 중요하다 (12장 비목표 9·10번).

예시 (문제·Wow 19/20 vs 에이전트 설계 9/25로 크게 갈린 제출물):

> **문제·Wow · 정하나**: "데모 시작하자마자 확신이 왔어요. 이 팀은 진짜 문제를 알고 있네요. 19점입니다."
>
> **에이전트 설계 · 이도윤**: "잠깐만요, 정하나 심사위원님. 그 '결정적 경험'을 만든 게 멀티에이전트 라우팅이 아니라 단일 프롬프트 호출 한 번이더라고요. `agent_orchestrator.py`에 에이전트 이름은 3개 붙어 있는데 서로 통신하는 코드는 없습니다. 저는 9점입니다."
>
> **정하나**: "설계가 단순한 것과 문제를 잘 짚은 건 다른 얘기죠. 제 점수는 유지합니다."
>
> **이도윤**: "동의합니다, 항목이 다르니까요. 다만 발표에서 '멀티에이전트 시스템'이라고 소개한 부분은 운영·품질 심사위원님도 참고하시면 좋겠네요."

트리거 조건: (a) 반복 채점 편차가 임계값을 넘거나, (b) 서로 연관된 두 항목의 점수가 크게 갈릴 때(예: 문제·Wow는 상위권, 에이전트 설계는 하위권). 새 API 호출을 늘리지 않도록, 이미 산출된 두 페르소나의 근거 텍스트를 대화체로 재구성하는 것으로 구현하고 9장의 보조 호출 예산 안에서 처리한다.

### 13.2 신뢰구간 표시 예시

항목별 점수는 평균값만이 아니라 반복 채점 값과 편차를 함께 노출한다.

```
에이전트 설계     18 / 25점   (2회 채점: 17, 19 — 편차 2점, 신뢰도 높음)
운영·품질         12 / 20점   (3회 채점: 8, 15, 13 — 편차 7점, 신뢰도 낮음 — 사람 심사위원 확인 권장)
```

규칙 (8.6과 동일 기준을 표시로 연결):

- 최대-최소 편차가 배점의 15% 이내 → "신뢰도 높음", 평균값 그대로 사용
- 3회차까지 갔는데도 15% 초과 → "신뢰도 낮음 - 사람 심사위원 우선 확인" 라벨을 텍스트로 명시
- 리더보드에서는 "신뢰도 낮음" 항목이 하나라도 있는 팀의 이름 옆에 "검토 필요" 텍스트를 붙여, 사람 심사위원이 우선적으로 볼 수 있게 한다 (색상/이모지가 아니라 텍스트 라벨로 표시).

### 13.3 애니메이션 리더보드 흐름

1. 페이지가 열리면 순위 없이 제출 순서대로 팀이 나열되고, 항목 게이지는 모두 비어 있다.
2. "채점 결과 공개" 버튼을 누르면(또는 자동 시작), 첫 항목(문제·Wow)부터 모든 팀의 게이지가 동시에 채워진다 (CSS transition, 약 1~1.5초).
3. 항목이 채워질 때마다 해당 항목 기준으로 팀 카드가 부드럽게 재정렬된다 - 아직 최종 순위가 아니라는 안내 문구를 함께 보여준다.
4. 5개 항목이 순서대로 공개된다: 문제·Wow → 에이전트 설계 → 동작·완성도 → 운영·품질 → 발표·협업.
5. 마지막에 총점 게이지가 채워지며 최종 순위로 정렬되고, 각 팀 카드에 해당하는 뱃지(13.4)가 순차적으로 나타난다.
6. 정지/재생 컨트롤을 제공해 발표 중 리와인드할 수 있게 한다. 이미 계산된 점수 데이터를 순서대로 재생하는 것뿐이므로 서버 재호출은 없다.

### 13.4 뱃지 상세 (페르소나와 연결)

각 뱃지는 해당 항목을 담당한 심사위원 페르소나가 "수여"하는 형태로 문구를 구성해, 페르소나와 뱃지가 하나의 서사로 이어지게 한다.

| 뱃지 | 부여 조건 | 리포트 문구 |
|---|---|---|
| 가장 대담한 도전상 | 문제·Wow 최고점 | "정하나 심사위원이 망설임 없이 최고점을 준 유일한 팀" |
| 설계 장인상 | 에이전트 설계 최고점 | "이도윤 심사위원의 까다로운 질문에도 코드로 답한 팀" |
| 제로 버그상 | 동작·완성도 만점 AND 콘솔 에러 0건 | "박세이 심사위원이 아무리 눌러봐도 안 깨진 유일한 데모" |
| 안전제일상 | 운영·품질 만점 | "최민석 심사위원의 장애 시나리오 질문을 다 통과한 팀" |
| 원팀상 | 발표·협업 항목에서 커밋 기여 분포가 가장 고름 | "한다은 심사위원이 커밋 로그에서 실제 팀워크를 확인한 팀" |
