# auto-as 심사 패널 고도화 PRD (v1)

작성 목적: 아래 3개 요구사항을 5인 팀이 나눠 구현할 수 있도록 task로 분해한다.
기존 시스템(정적분석→브라우저→규칙채점→5인 패널 토론→HTML 리포트/리더보드)은 유지하되,
(1) 페르소나를 재설계하고 (2) 채점 기준별 역할 분담·토론 구조를 개선하고 (3) 근거·리포트 가독성을 검증한다.

## 0. 토대: 공식 루브릭 교체 (선행 작업, 모든 task의 전제)

대회 공식 루브릭이 코드에 박힌 값과 다르다. 아래 신규 루브릭으로 교체한다.

| 항목 | 신규 배점 | 정의 | 근거 소스 |
|---|---|---|---|
| 문제·Wow | 20 | 해결할 문제를 명확히 정의했고 서비스만의 차별화된 가치가 있는가 | 데모 실행 결과, README/피치, 시나리오 |
| AI 기능 구현 | 20 | 적절한 도구·RAG 등 기술로 필요한 기능을 효과적으로 구현했는가 | 소스 코드(도구/RAG/오케스트레이션) |
| 동작 완성도 | 25 | 정의한 문제가 실질적으로 해결됐고 주요 기능이 안정적으로 구현됐는가 | Playwright 실행 결과 + 시나리오 성공률 + 콘솔 에러 |
| 운영 품질 | 15 | 골든 데이터셋을 구축하고 이를 활용해 서비스 성능·품질을 적절히 평가했는가 | 소스 코드(eval 스크립트, 골든셋, 평가 지표) |
| 발표·협업 | 20 | 발표 스토리·전달 구조가 명확하고 팀 역할 분담·협업이 체계적인가 | 발표 자료, git 로그(작성자 분포·시점) |

변경점(기존 대비):
- "에이전트 설계 25" -> "AI 기능 구현 20" (이름·배점 변경)
- "동작·완성도 20" -> 25 (최고 배점)
- "운영·품질 20" -> 15, 그리고 의미가 "eval/모니터링/가드레일"에서 "골든 데이터셋 구축 + 성능/품질 평가"로 좁혀짐
- "발표·협업 15" -> 20

영향 파일: scoring.py(RUBRIC, 각 항목 채점식), panel.py(max_score/role), pipeline.py(SIGNALS의 운영품질 신호를 골든셋/eval 중심으로 재정의), leaderboard.py(뱃지 문구), report.py(라벨). 배점은 설정으로 조정 가능하게 유지한다.

가드레일(불변): PRD 12장 비목표 10 — 페르소나 성격이 루브릭 근거 없이 점수를 가감하지 않는다. 모든 점수에는 evidence/references가 붙는다. 저장소 코드는 실행하지 않는다.

## 1. 요구사항 A — 페르소나 5종 재설계 및 개성 강화

목표: 심사위원 5명을 전문 분야·말투·판단 기준이 뚜렷한 캐릭터로 재정의한다.
실존 인물 이름은 패러디 모티프로만 차용하고 실제 발언·인격은 재현하지 않는다.

기본값(조정 가능) 페르소나 5종 및 담당:
- 아이유(VC 투자자): 문제·Wow
- 박명수(100만 IT 유튜버): 발표·협업
- 손흥민(오픈소스 개발자): AI 기능 구현
- 강호동(FAANG Staff Engineer): 동작 완성도
- 백종원(논문 리뷰어): 운영 품질(골든셋/eval)

산출물:
- 최종 페르소나 5종 목록
- 페르소나별 설정 문서(docs/judge-personas.md 갱신): 역할·전문 분야, 말투·표현 방식, 선호·비판 성향, 중요하게 보는 기준, 대표 발언 예시
- 페르소나별 프로필 이미지 1장(총 5장) — leaderboard.html 소개 카드에 적용
- panel.py의 PERSONAS 딕셔너리 및 프롬프트 반영
- 페르소나 관련 테스트 또는 샘플 출력(test_smoke.py 갱신)

## 2. 요구사항 B — 채점 기준별 역할 분담 및 3단계 토론 개선

핵심 변경: 각 페르소나는 5개 기준 중 자신과 연관성 높은 2개 criteria에만 의견을 낸다.
동일 criteria를 2명이 담당하게 해서 관점 충돌을 유도한다.

기본값(조정 가능) 담당 매핑 — 대칭형 2:2(각 기준=2명, 각 페르소나=2기준):
- 문제·Wow: 아이유(주), 박명수(부)
- AI 기능 구현: 손흥민(주), 백종원(부)
- 동작 완성도: 강호동(주), 손흥민(부)
- 운영 품질: 백종원(주), 강호동(부)
- 발표·협업: 박명수(주), 아이유(부)
- 코디네이터: 점수 확정을 담당하는 별도 역할(채점 페르소나 아님)

30초 3단계 토론 구조(기존 build_discussion / run_openai_panel 재구성):
1) 1차 채점 초안: 각 심사위원이 담당 criteria에 대해 근거와 함께 독립 점수 제시
2) 의견 조정: 동일 criteria 담당 2명이 서로의 점수·근거를 비교, 캐릭터 말투로 반박·재반박
3) 코디네이터 최종 결정: 각 근거를 검토해 최종 점수와 이유 정리. 근거 없는 점수 조정 금지.

산출물:
- 페르소나별 담당 criteria 매핑표(코드 상수 + 문서)
- 30초 채점 대화 흐름(단계별 타임라인)
- 1차 채점 프롬프트 / 의견 조정 프롬프트 / 코디네이터 최종 결정 프롬프트 (local 규칙판 + OpenAI 프롬프트판)
- 페르소나별 말투·갈등 표현 규칙(docs/dialogue-quality.md 확장)
- 실제 샘플 채점 대화 1~2건(examples 기반 재생)
- 최종 점수와 근거가 일관되게 연결되는지 확인하는 테스트

## 3. 요구사항 C — 채점 근거 및 리포트 가독성 검증

목표: 각 조 report.html을 실제 샘플 제출물에 적용해 근거의 명확성을 검증하고 개선한다.

검증 항목:
- 점수마다 구체적 근거가 연결되어 있는가
- 근거가 실제 정적 분석/브라우저 실행 결과에 기반하는가
- 심사위원 의견과 최종 점수 사이 논리적 연결이 있는가
- 리포트 하단이 읽기 쉽고 핵심을 빠르게 파악할 수 있는가
- 같은 근거가 여러 criteria에 중복 사용되지는 않는가
- 근거가 부족한 경우 이를 명시하는가

산출물:
- 기존 조별 산출물 샘플(examples/submissions 활용) 및 샘플별 생성 report
- 샘플 채점 결과 및 검증 의견(문서)
- 리포트 가독성 개선안 + criteria별 근거 표시 형식
- 근거 누락·중복·불일치 검증 체크리스트(자동 점검 스크립트/함수)
- 개선된 HTML report 샘플
- 검증 결과를 반영한 테스트 코드

## 4. 통합 목표

페르소나 설정 -> 기준별 의견 제시 -> 토론·조정 -> 코디네이터 최종 결정 -> 근거 기반 HTML 리포트
흐름이 끝까지 자연스럽게 이어지는지 end-to-end로 확인한다.

## 5. 5인 팀 오너십(기본 제안)

- 팀원1(페르소나 오너): 요구사항 A 전반 — 페르소나 정의·문서·이미지·panel.py 반영
- 팀원2(패널 엔진 오너): 요구사항 B의 매핑 상수 + 3단계 토론 구조(local/openai)
- 팀원3(스코어링/코디네이터 오너): 루브릭 교체(0번) + 다대다 채점 반영 + 코디네이터 최종 결정 로직 + 근거 일관성
- 팀원4(리포트/리더보드 오너): report.html/leaderboard.html 가독성 개선 + 이미지 적용 + criteria별 근거 표시
- 팀원5(검증/QA 오너): 샘플 채점 실행 + 근거 누락·중복·불일치 체크리스트 + 테스트 코드 + e2e 확인

## 6. 검증 기준

- python -m pytest test_smoke.py 통과(HTML 구조 검증 포함)
- 기본 모드 local_provisional는 외부 키 없이 결정론적으로 동작
- 모든 점수에 evidence/references 존재, 근거 없는 점수 조정 없음
